# Agent Brief template

An **Agent Brief** is the durable contract between the planning phase (`/scope`)
and the execution phase (`/execute`). It is posted as a comment on the issue/
ticket. The issue body becomes historical context; the brief is the spec a
runner implements against and a reviewer checks against.

The single rule that makes a brief useful: **every acceptance criterion must be
verifiable** — concrete enough that the runner can check its own answer when
done. "Produces output X for input Y" is verifiable. "Add the scheduling
feature" is not.

Each acceptance criterion carries an `[AFK]` or `[HITL]` prefix, decided here at
brief-writing time (see `../docs/concepts.md` for the AFK/HITL rubric):

- `[AFK]` (Away From Keyboard) — an agent can do it and verify it without you.
- `[HITL]` (Human In The Loop) — needs your sign-off before it counts.

---

## Schema

```markdown
> *This brief was generated by AI. Review before relying on it.*

## Agent Brief

**Category:** bug | enhancement
**Risk level:** 1 | 2 | 3        (1 = docs/config, 2 = tooling, 3 = sensitive/critical path)
**Blocked by:** #N | None         (inter-ticket dependency, so a conductor can order tickets)

**Summary:** one sentence — what this ticket delivers.

**Current behavior:** what happens today (for a bug: the exact wrong behavior + how to trigger it).

**Desired behavior:** what should happen, and the source of truth for "correct".

**Key interfaces:** the modules / functions / endpoints / files the change touches,
named by identifier rather than line number (line numbers rot). For a migration,
cite the proven pattern to clone. For a bug, cite the suspected surface + the
repro loop.

**Acceptance criteria:**
- [ ] [AFK] <verifiable criterion — runner can self-check>
- [ ] [AFK] <verifiable criterion>
- [ ] [HITL] <criterion that needs human sign-off, and why>

**Out of scope:** explicit list of things NOT to touch — adjacent cleanup,
speculative refactors, deferred sub-features. The reviewer fails the diff if it
touches anything here.
```

## Worked example

```markdown
> *This brief was generated by AI. Review before relying on it.*

## Agent Brief

**Category:** enhancement
**Risk level:** 2
**Blocked by:** None

**Summary:** Add a `--json` flag to the `report` command so output can be piped to other tools.

**Current behavior:** `report` prints a human-formatted table only; there is no machine-readable output.

**Desired behavior:** `report --json` prints a JSON array of the same rows; the default (no flag) is unchanged.

**Key interfaces:** `cli/report.py::run()` (arg parsing + dispatch), `cli/format.py::to_table()`
(mirror this to add `to_json()`), existing tests in `tests/test_report.py`.

**Acceptance criteria:**
- [ ] [AFK] `report --json` emits valid JSON (parses with `json.loads`) with one object per row
- [ ] [AFK] `report` with no flag produces byte-identical output to before (golden test passes)
- [ ] [AFK] a new test covers the `--json` path
- [ ] [HITL] confirm the JSON field names match what the downstream dashboard expects

**Out of scope:** CSV output, changing the table format, refactoring `format.py` beyond adding `to_json()`.
```
